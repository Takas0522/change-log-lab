### Critical問題（5件）

#### C-001: トレーサビリティマトリクスの具体性不足
**ISO 29148基準**: 追跡可能性  
**該当箇所**: 7.1 ビジネス要件→システム要件マッピング  
**問題内容**:  
- ビジネス要件（BR-001～BR-007）が7.1のテーブルで参照されているが、セクション2では明示的に定義されていない
- テーブルには「BR-001: タスク管理の効率化」のような記述があるが、2.1節には対応する要件IDがない
- トレーサビリティが不完全で、ビジネス価値と機能要件の紐付けが追跡できない

**修正提案**:
```markdown
### 2.1 ビジネス要件（Business Requirements）

**ビジネス価値**: 作業効率30%向上、タスク完了率20%向上
**受け入れ基準**: 
- ToDo作成から完了まで5クリック以内で完結する
- タスク登録時間が従来の50%削減される

#### BR-002: 柔軟な分類機能
（以下同様に定義）
```

#### C-004: データモデルの完全性不足
**ISO 29148基準**: 完全性  
**該当箇所**: 8.1 エンティティ定義  
**問題内容**:
- ToDoエンティティに削除フラグがない（REQ-FUNC-005で削除機能があるが物理削除か論理削除か不明）
- インデックス定義が存在しない（性能要件REQ-PERF-002を満たせない可能性）
- 外部キー制約の削除時動作（CASCADE / SET NULL / RESTRICT）が定義されていない

**修正提案**:
```markdown
#### ToDo エンティティ
| 項目名 | データ型 | 制約 | 説明 |
|---|---|---|---|
| ToDoID | BIGINT | PK, AUTO INCREMENT | ToDo識別子 |
| Title | NVARCHAR(200) | NOT NULL | タイトル |
| Content | NVARCHAR(MAX) | NULL | 内容（最大5000文字をアプリケーション層で制御） |
| Status | VARCHAR(20) | NOT NULL, DEFAULT '未着手', CHECK(Status IN ('未着手','着手中','完了','放棄')) | ステータス |
| IsDeleted | BIT | NOT NULL, DEFAULT 0 | 論理削除フラグ |
| CreatedAt | DATETIME2 | NOT NULL, DEFAULT GETUTCDATE() | 投稿日時（UTC） |
| UpdatedAt | DATETIME2 | NOT NULL, DEFAULT GETUTCDATE() | 更新日時（UTC） |

**インデックス定義**:
- IX_ToDo_Status_CreatedAt (Status, CreatedAt DESC) - 一覧表示の高速化
- IX_ToDo_IsDeleted_CreatedAt (IsDeleted, CreatedAt DESC) - 削除済み除外クエリの高速化

**外部キー制約**:
- ToDoLabel.ToDoID → ToDo.ToDoID (ON DELETE CASCADE)
- ToDoLabel.LabelID → Label.LabelID (ON DELETE CASCADE)
```

### Major問題（8件）

#### M-001: ステータス定義の不一致
**ISO 29148基準**: 一貫性  
**該当箇所**: 
- 1.3 定義: 「未着手、着手中、完了、放棄」
- REQ-FUNC-006: 「未着手 / 着手中 / 完了 / 放棄」
- データモデル: 「DEFAULT '未着手'」（日本語）
**問題内容**:
- データベースで日本語ステータスを使用すると、国際化時に問題が発生
- データ型がVARCHAR(20)では将来の追加に柔軟性がない

**修正提案**:
```markdown
### 1.3 定義
| ステータス | コード値 | 表示名 | 説明 |
|---|---|---|---|
| Not Started | NOT_STARTED | 未着手 | タスク未着手 |
| In Progress | IN_PROGRESS | 着手中 | タスク作業中 |
| Completed | COMPLETED | 完了 | タスク完了 |
| Abandoned | ABANDONED | 放棄 | タスク放棄 |

データベース定義:
Status VARCHAR(20) NOT NULL DEFAULT 'NOT_STARTED' CHECK(Status IN ('NOT_STARTED','IN_PROGRESS','COMPLETED','ABANDONED'))
```

#### 3.1.5 エラーハンドリング

#### M-004: ページネーション仕様の曖昧さ
**ISO 29148基準**: 明確性  
**該当箇所**: REQ-FUNC-002 ToDo一覧表示  
**問題内容**:
- ページネーション方式（オフセット方式 / カーソル方式）が不明
- 最大表示件数の上限がない（パフォーマンスリスク）
- ページ遷移時のソート順維持の仕様がない

**修正提案**:
```markdown
##### REQ-FUNC-002: ToDo一覧表示（修正版）
- **入力**: 
  - ページ番号（整数、1以上、デフォルト1）
  - 1ページあたりの表示件数（整数、10-100、デフォルト20）
  - ソートキー（createdAt / updatedAt / title、デフォルト: createdAt）
  - ソート方向（asc / desc、デフォルト: desc）
  - 絞込条件（任意）
- **処理**: 
  - オフセット方式のページネーション（OFFSET/LIMIT）を使用
  - ソート条件とページ番号はURLクエリパラメータで管理
- **出力**: 
  - ToDoリスト
  - ページネーション情報（currentPage, totalPages, totalCount, hasNext, hasPrevious）
- **受け入れ基準**: 
  - 表示件数は10-100の範囲で指定可能
  - 範囲外の場合は400 Bad Requestを返却
  - 存在しないページ番号の場合は空リストを返却（エラーにしない）
```

#### M-008: 削除機能の仕様矛盾
**ISO 29148基準**: 一貫性  
**該当箇所**: REQ-FUNC-005, REQ-FUNC-011  
**問題内容**:
- REQ-FUNC-005「削除されたToDoは復元不可」は物理削除を示唆
- REQ-REL-002「データ損失ゼロ」と矛盾
- 監査ログやデータ分析のニーズが考慮されていない

**修正提案**:
```markdown
##### REQ-FUNC-005: ToDo削除（修正版）
- **説明**: ユーザーは不要なToDoを論理削除できる
- **処理**: 
  - 削除確認ダイアログを表示
  - ユーザーの確認後、IsDeletedフラグをtrueに設定
  - 一覧表示では論理削除されたToDoを非表示
- **出力**: 
  - 削除成功メッセージ
  - 一覧画面へリダイレクト
- **受け入れ基準**: 
  - 論理削除後、一覧画面から該当ToDoが見えなくなる
  - データベースからは削除されず、IsDeleted=1として保持される
  - 管理機能で削除済みToDoの参照が可能（将来実装）

同様にREQ-FUNC-011（ラベル削除）も論理削除に変更を推奨
```

### ISO 29148品質基準の解説

| 基準 | 説明 | 本レビューでの評価 |
|---|---|---|
| 明確性 | 各要件は一意に解釈可能か | △ ステータス定義、日時扱いに曖昧さあり |
| 完全性 | 必要な情報が揃っているか | △ エラーハンドリング、データ制約が不足 |
| 一貫性 | 要件間に矛盾がないか | △ 削除機能とデータ損失防止が矛盾 |

| 追跡可能性 | 上位要件と紐づいているか | ✗ ビジネス要件の定義不足 |
| 実現可能性 | 技術的に実現可能か | △ 検索性能に懸念あり |

**Minor**: 推奨される改善事項、読みやすさの向上
