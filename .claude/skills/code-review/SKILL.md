# Code & Document Review Skill

コード・ドキュメントを体系的にレビューし、構造化されたフォーマットでフィードバックを提供するスキル。
本リポジトリのマイクロサービス構成（ASP.NET Core + Angular + PostgreSQL）に最適化されている。

## When to Use This Skill

- **コードレビュー**: C#、TypeScript/Angular、SQL のソースコードの品質・安全性を評価
- **PR / MR レビュー**: プルリクエストの変更差分に対するレビューコメントの生成
- **ドキュメントレビュー**: 要求仕様書（SRS）、設計書（SDD）、API 仕様の品質・整合性の検証
- **設計レビュー**: アーキテクチャ・データモデル・API 設計の妥当性評価
- **品質監査**: 既存コードベースの品質状態のスナップショット評価
- **リファクタリング前の評価**: 改善対象の特定と優先順位付け

## レビュー原則

1. **指摘と修正案はセット**: 問題の指摘には必ず具体的な修正案を併記する
2. **重大度の明示**: すべての指摘に Critical / Major / Minor のラベルを付与する
3. **良い実装の承認**: 問題点だけでなく、優れた実装パターンも積極的に評価する
4. **実行可能な単位**: 1 レビューあたりの指摘は実行可能な単位にまとめる
5. **再現可能性**: 問題の再現手順や該当箇所を明確に示す
6. **日本語で記述**: すべてのレビューコメントは日本語で記述する

## レビューワークフロー

### Phase 1: コンテキスト収集

1. レビュー対象ファイルの全文を読み取る
2. 関連する依存ファイル（インターフェース、モデル、DTO）を特定・読み取る
3. 呼び出し元・呼び出し先を調査し影響範囲を把握する
4. 関連する仕様書・設計書があれば参照する

### Phase 2: レビュー実施

レビュー対象の種類に応じて、該当するチェックリストを適用する。

- **コードレビュー**: [コードレビューチェックリスト](./references/code-review-checklist.md) を適用
- **ドキュメントレビュー**: [ドキュメントレビュー基準](./references/document-review-standards.md) を適用

### Phase 3: レポート生成

収集した指摘事項をレビューレポートフォーマットに整形して出力するか
ユーザーからの指定がない場合は結果を返答する。

## スコアリング基準

### 総合評価の算出

```
基本スコア = 100

減点:
  🔴 Critical × (-20点)
  🟡 Major    × (-5点)
  🔵 Minor    × (-1点)

最低スコア = 0
```

| スコア | 評価 | 判定 |
|--------|------|------|
| 90–100 | **A** — Excellent | リリース可。軽微な改善のみ |
| 75–89  | **B** — Good | Major 以上を修正してリリース |
| 50–74  | **C** — Needs Work | Critical / Major の修正必須 |
| 0–49   | **D** — Critical | リリース不可。即時対応が必要 |

## コードレビュー観点（概要）

詳細なチェックリストは [references/code-review-checklist.md](./references/code-review-checklist.md) を参照。

### 共通（すべてのコード）

| 観点 | 説明 | 重大度目安 |
|------|------|-----------|
| 正確性 | 仕様に対するロジックの正しさ、境界値・例外ケースの網羅 | Critical–Major |
| セキュリティ | 入力検証、認証・認可、秘匿情報、インジェクションリスク | Critical |
| エラーハンドリング | 例外設計、ログ出力、グローバルハンドリング原則の遵守 | Major |
| パフォーマンス | N+1 問題、不要な同期 I/O、メモリ効率 | Major |
| 設計・保守性 | SRP、命名一貫性、DRY、循環依存 | Major–Minor |
| 並行性 | 共有可変状態、競合条件、デッドロックリスク | Critical–Major |
| テスト | ユニット / 統合テストの有無と妥当性 | Major |
| ドキュメント | Public API の JSDoc / XML ドキュメントコメント | Minor |

### C# 固有

- `async/await` 原則（`.Result` / `.Wait()` の禁止、`CancellationToken` 伝播）
- Nullable 参照型の適切な使用
- グローバル例外ハンドリング（各所での try/catch は原則不要）
- DI ライフタイム（Singleton / Scoped / Transient）の整合性
- EF Core: `AsNoTracking`、N+1、トランザクション管理
- 構造化ログ（`ILogger` プレースホルダー形式）

### Angular 固有

- スタンドアロンコンポーネント（NgModules 不使用）
- `input()` / `output()` 関数（デコレータではなく）
- Signals による状態管理、`computed()` による派生状態
- ネイティブ制御フロー（`@if` / `@for` / `@switch`）
- `inject()` 関数による DI（コンストラクタインジェクション不使用）
- OnPush 変更検知
- メモリリーク防止（`async` パイプ / `takeUntil`）

### SQL 固有

- パラメータ化クエリ（動的 SQL 連結の禁止）
- インデックスの妥当性、`SELECT *` の禁止
- トランザクション分離レベルの適切性
- 外部キー / 一意制約 / チェック制約の適用

## ドキュメントレビュー観点（概要）

詳細な基準は [references/document-review-standards.md](./references/document-review-standards.md) を参照。

### 品質基準

| 観点 | 説明 |
|------|------|
| 正確性 | 技術的な記述の正しさ、既存実装との整合性 |
| 完全性 | 必要なセクション・項目の網羅 |
| 一貫性 | 用語・表記・フォーマットの統一 |
| 明確性 | 曖昧な表現の排除（「等」「適宜」の回避） |
| 追跡可能性 | 要件 ID の付与、上位・下位文書との紐付け |
| 検証可能性 | 各要件が検証可能な形で記述されているか |

### ISO 準拠確認

- ISO/IEC/IEEE 29148（要求仕様）
- IEEE 1016-2009（設計記述）
- ISO/IEC/IEEE 15289（文書内容）
