# ドキュメントレビュー基準

本基準は、要求仕様書・設計書・API 仕様等のドキュメントをレビューする際に適用する品質基準を定義する。

---

## 適用規格

すべてのドキュメントは以下の ISO 規格に準拠する形式・構造で作成されていることを確認する。

| 規格 | 説明 | 対象文書 |
|------|------|---------|
| ISO/IEC/IEEE 12207 | ソフトウェアライフサイクルプロセス | 全般 |
| ISO/IEC/IEEE 15288 | システムライフサイクルプロセス | 全般 |
| ISO/IEC/IEEE 29148 | 要求エンジニアリング（要求仕様書の形式） | SRS |
| ISO/IEC/IEEE 15289 | ライフサイクル情報成果物（文書内容） | 全般 |
| IEEE 1016-2009 | ソフトウェア設計記述 | SDD |

---

## 共通文書フォーマット確認

すべてのドキュメントに以下のセクションが含まれていることを確認する。

### 必須セクション

1. **表紙**
   - [ ] 文書タイトル
   - [ ] 文書番号とバージョン
   - [ ] 作成日 / 更新日
   - [ ] 作成者 / 承認者

2. **変更履歴**
   - [ ] バージョン、日付、変更内容、担当者が表形式で記録されている

3. **目次**
   - [ ] セクション構成を反映した目次が存在する

4. **1. はじめに**
   - [ ] 1.1 目的
   - [ ] 1.2 スコープ
   - [ ] 1.3 用語と定義
   - [ ] 1.4 参照文書

5. **2. 本文**（文書種別に応じた内容）

6. **付録**（必要に応じて）

## 品質観点チェックリスト

### 正確性（Accuracy）

- [ ] 技術的な記述が正しく、既存実装と整合している
- [ ] 数値・バージョン・日付等の具体的情報が最新である
- [ ] コード例やコマンド例が正しく動作する
- [ ] 図表の内容とテキストの説明が一致している
- [ ] 外部参照（URL、文書番号）が有効である

### 完全性（Completeness）

- [ ] 必要なセクション・項目がすべて記載されている
- [ ] 「TBD」「TODO」等の未完成マーカーが残っていない
- [ ] すべての機能要件・非機能要件が漏れなく記述されている
- [ ] エラーケース・例外ケースの記述がある
- [ ] 前提条件・制約事項が明記されている

### 一貫性（Consistency）

- [ ] 用語の定義と使用が文書全体で統一されている
- [ ] 略語の初出時に正式名称が記載されている（例: SRS (Software Requirements Specification)）
- [ ] 表記ルール（全角/半角、句読点、見出しレベル）が統一されている
- [ ] 番号体系（REQ-001、SYS-001 等）が一貫している
- [ ] 図表の番号・キャプション形式が統一されている

### 明確性（Clarity）

- [ ] 曖昧な表現がない（「等」「適宜」「必要に応じて」「原則として」の回避）
- [ ] 一つの文が一つの要件のみを記述している（複合要件の分離）
- [ ] 主語と述語が明確である
- [ ] 略語・専門用語が初出時に定義されている
- [ ] 否定表現の二重使用がない

### 追跡可能性（Traceability）

- [ ] すべての要件・仕様に一意の識別子が付与されている（例: REQ-001, SYS-001）
- [ ] 上位文書（要求仕様）と下位文書（設計書）の紐付けがある
- [ ] テストケースとの対応関係が示されている（該当箇所）
- [ ] 変更履歴から要件の変遷を追跡できる

### 検証可能性（Verifiability）

- [ ] 各要件が客観的に検証可能な形で記述されている
- [ ] 定量的な基準が示されている（「高速」ではなく「応答時間 200ms 以内」）
- [ ] 受入基準（Acceptance Criteria）が明確に定義されている
- [ ] テスト方法・検証手順が推測可能である

## 文書種別固有のチェックリスト

### 要求仕様書（SRS）— ISO/IEC/IEEE 29148

- [ ] 機能要件と非機能要件が明確に分離されている
- [ ] 各要件に優先度（Must / Should / Could / Won't）が設定されている
- [ ] ユースケースまたはユーザーストーリーが記述されている
- [ ] インターフェース要件（外部システム連携、API）が定義されている
- [ ] セキュリティ要件が明記されている
- [ ] パフォーマンス要件が定量的に記述されている
- [ ] 可用性・信頼性要件が定義されている
- [ ] データ要件（保持期間、バックアップ等）が記述されている
- [ ] 制約条件（技術制約、運用制約）が明記されている

### 設計書（SDD）— IEEE 1016-2009

- [ ] アーキテクチャ概要（コンポーネント図、配置図等）が記述されている
- [ ] 各コンポーネントの責務と境界が明確に定義されている
- [ ] コンポーネント間のインターフェース（API 仕様、メッセージ形式）が定義されている
- [ ] データモデル（ER 図、テーブル定義）が記述されている
- [ ] シーケンス図等で主要な処理フローが説明されている
- [ ] エラーハンドリング戦略が記述されている
- [ ] セキュリティ設計（認証・認可モデル）が記述されている
- [ ] 設計判断の根拠（Design Rationale）が記録されている

### API 仕様書

- [ ] すべてのエンドポイントが一覧化されている
- [ ] 各エンドポイントの HTTP メソッド、パス、説明が記述されている
- [ ] リクエスト / レスポンスの JSON スキーマが定義されている
- [ ] ステータスコードとエラーレスポンスの形式が定義されている
- [ ] 認証・認可の要件が各エンドポイントに記述されている
- [ ] リクエスト / レスポンスの具体例が示されている
- [ ] レート制限・ページネーション等の運用仕様が記述されている

## 重大度の判定基準

### 🔴 Critical（即時修正必須）

- 技術的に誤った記述が含まれており、そのまま実装すると障害に繋がる
- セキュリティ要件が欠落しており、脆弱性を生む可能性がある
- 要件の矛盾・重複により実装判断が不可能
- 必須セクションが丸ごと欠落している

### 🟡 Major（次版までに修正）

- 要件の記述が曖昧で、実装者の解釈により結果が異なる可能性がある
- 非機能要件（パフォーマンス、可用性等）の定量基準が欠如している
- 追跡可能性が確保されていない（ID 未付与、上下位文書の紐付けなし）
- 図表と本文の説明に不整合がある
- 用語の不統一が散見される

### 🔵 Minor（改善推奨）

- 表記ルールの不統一（全角/半角、句読点等）
- 冗長な記述・不要な繰り返し
- 図表の番号・キャプションの形式不統一
- 参考情報や補足説明の不足
- 文法・誤字脱字
